01 — visão geral

O que estamos construindo

O Broto CMS é uma plataforma de produto da Brototec — não apenas um CMS para clientes, mas uma infraestrutura que serve tanto sites institucionais quanto sistemas customizados, tudo no mesmo ecossistema técnico.

🌱
A ideia central Um monorepo com Next.js + Payload CMS rodando juntos. O front-end consome a API do Payload. Clientes editam pelo painel do Broto CMS. Você cria temas em código e pluga no sistema como Drupal faz com themes.

O produto serve dois mercados

Mercado 1
Sites e landing pages — clientes que querem autonomia para editar conteúdo via Broto CMS, sem depender de dev.
Mercado 2
Sistemas customizados — CRUDs completos para operações internas, painéis de gestão, plataformas específicas de negócio.
Núcleo comum
A mesma stack — Payload cuida da base de dados, auth, upload e API. O front Next.js consome tudo. Só muda o "tema" e as collections.
02 — stack

Decisões técnicas e por quê

Tecnologia Papel Por que esta escolha Alternativa descartada
Next.js 15 Framework full-stack App Router, RSC, SSR, API Routes — tudo incluso. Você já conhece React. Remix (menos ecossistema)
Payload CMS v3 CMS + Admin + API Roda dentro do Next.js. TypeScript nativo. Collections em código. Open source. Strapi (mais pesado, menos TS)
TypeScript Linguagem Tipos compartilhados entre CMS e front. Menos bug. Você já usa no front. JavaScript puro (sem segurança)
PostgreSQL Banco de dados Robusto, relacional, suportado pelo Payload nativamente. Escala real. MongoDB (menos estruturado)
Tailwind CSS v4 Estilização Você já domina. Funciona perfeitamente com Next.js e Server Components. Styled Components (runtime CSS)
Neon / Supabase Postgres na nuvem (free) Postgres gerenciado grátis para começar. Migra pra VPS quando crescer. PlanetScale (não é Postgres)
Vercel Deploy (free tier) Deploy automático por push. Domínio grátis. Zero config com Next.js. Netlify (menos integrado)
Uploadthing / S3 Upload de arquivos Payload tem adapter nativo. Uploadthing é grátis para começar. Cloudinary (vendor lock-in)
💡
Sobre Node.js vs Python Como front-end com experiência em JS/TS, Node.js é a escolha certa. Você não aprende uma nova linguagem — aprende um novo contexto. Os tipos TypeScript são compartilhados entre o Payload (backend) e o Next.js (front). Isso vale ouro.
03 — princípios

Princípios técnicos do Broto CMS

Collections em código
Cada tipo de conteúdo é definido em TypeScript. Versionado no git. Nunca num banco de dados opaco.
Temas modulares
Um tema é uma pasta com componentes React + config. Plugável como Drupal/WordPress. Clientes trocam de tema sem perder conteúdo.
API-first
Todo conteúdo exposto via REST API automática do Payload. Front consome via fetch. Amanhã pode virar app mobile sem reescrever backend.
Zero lock-in
Código do cliente é do cliente. Self-hosted. Sem dependência de plataforma fechada. Coerência com a filosofia da Brototec.
Monorepo único
CMS + front + admin no mesmo repositório. Tipos compartilhados. Deploy único. Menos complexidade operacional.

05 — collections

Collections do Broto CMS

Collections são os tipos de conteúdo do seu CMS. Definidos em TypeScript, versionados no git, aparecem automaticamente no painel admin.

Pages
Páginas do site com layout por blocos. Slug automático, SEO nativo.
title slug layout[] (blocks) meta status
Posts
Blog, notícias, cases. Com autor, categorias e rich text.
title content (rich text) author categories featuredImage publishedAt
Media
Upload centralizado com resize automático e alt text.
filename mimeType alt sizes (auto)
Users
Auth nativo do Payload. Roles: admin, editor, viewer.
email password (hashed) role name
Forms
Formulários dinâmicos com campos configuráveis pelo admin.
title fields[] submitLabel redirect
FormSubmissions
Respostas dos formulários, vinculadas ao form original.
form (relation) submissionData[] createdAt

Para sistemas customizados — collections extras

Quando o projeto é um sistema (não apenas um site), você adiciona collections de negócio:

Products / Services
Catálogo de produtos ou serviços com preços, variantes e imagens.
name price variants[] images[] category
Orders / Records
CRUD de registros operacionais — pedidos, contratos, agendamentos.
reference status customer items[] total
06 — temas

Sistema de temas modulares

Como o Drupal ou WordPress, um "tema" no Broto CMS é uma pasta autocontida com seus componentes React e configuração. O conteúdo vive no banco — o tema só decide como renderizar.

src/themes/broto-default/theme.config.ts typescript 
// Configuração do tema — exporta componentes para cada bloco
import type { ThemeConfig } from '@/lib/theme-types'

import { HeroBlock } from './components/HeroBlock'
import { ContentBlock } from './components/ContentBlock'
import { CTABlock } from './components/CTABlock'
import { Header } from './components/Header'
import { Footer } from './components/Footer'

export const theme: ThemeConfig = {
  name: 'broto-default',
  version: '1.0.0',
  
  // Layout global
  layout: { Header, Footer },
  
  // Mapa bloco → componente React
  blocks: {
    hero: HeroBlock,
    content: ContentBlock,
    cta: CTABlock,
  },
  
  // Fontes e tokens de design
  fonts: ['Sora', 'JetBrains Mono'],
  tokens: {
    primary: '#7ab648',
    background: '#f5f0e8',
  }
}
src/app/(frontend)/[slug]/page.tsx typescript 
// Renderizador dinâmico de páginas — tema plugável
import { getPayloadClient } from '@/lib/payload'
import { theme } from '@/themes/broto-default/theme.config'

export default async function Page({ params }) {
  const payload = await getPayloadClient()
  
  const page = await payload.find({
    collection: 'pages',
    where: { slug: { equals: params.slug } }
  })
  
  // Renderiza cada bloco com o componente do tema ativo
  return (
    <div>
      {page.docs[0].layout.map((block) => {
        const Block = theme.blocks[block.blockType]
        return <Block key={block.id} {...block} />
      })}
    </div>
  )
}
🎨
Como criar um tema para um cliente Duplica a pasta broto-default, renomeia para cliente-nome, ajusta cores/fontes/componentes. O conteúdo no banco não muda — só a apresentação visual. Você entrega o tema como parte do projeto.
07 — api

Camada de API

O Payload gera automaticamente uma REST API completa para cada collection. Você não escreve endpoints CRUD — eles existem por padrão.

REST API automática — exemplos de uso http 
# Listar páginas publicadas
GET  /api/pages?where[status][equals]=published

# Buscar página por slug
GET  /api/pages?where[slug][equals]=sobre

# Criar novo post (autenticado)
POST /api/posts
     Authorization: Bearer {token}
     Content-Type: application/json

# Upload de mídia
POST /api/media
     Content-Type: multipart/form-data

# Login e obter token
POST /api/users/login
     { "email": "...", "password": "..." }

# Endpoint customizado (você cria quando precisar)
GET  /api/custom/dashboard-metrics
⚡
Local API — zero latência no Next.js Dentro do Next.js, você chama payload.find() diretamente — sem HTTP. É uma chamada de função, não uma requisição de rede. Isso torna o SSR extremamente rápido.
08 — implementação

Roteiro de implementação

Ordem de aprendizado e construção — do setup inicial ao produto Brototec em produção.

FASE 0 Setup do ambiente ~2h
  • Instalar Node.js LTS (v20+) e pnpm: npm install -g pnpm
  • Criar projeto com CLI do Payload: npx create-payload-app@latest broto-cms
  • Escolher template: website → Next.js → TypeScript → PostgreSQL
  • Criar banco Postgres gratuito no Neon.tech (5GB free, sem cartão)
  • Configurar .env.local com a string de conexão do Neon
  • Rodar localmente: pnpm dev → acessar localhost:3000/admin
  • Criar usuário admin e explorar o painel gerado
FASE 1 MVP — primeiro site com CMS ~1 semana
  • Estudar o arquivo payload.config.ts gerado — entender a estrutura
  • Criar collection Pages com campos: title, slug, status, layout (blocks)
  • Criar collection Media para uploads
  • Criar collection Users com roles (admin / editor)
  • Criar 3 blocos básicos: HeroBlock, ContentBlock, CTABlock
  • Criar rota dinâmica /[slug]/page.tsx que renderiza os blocos
  • Estilizar com Tailwind — criar o tema broto-default
  • Testar criação de página pelo admin e visualizar no front
FASE 2 Produto — Broto CMS completo ~2 semanas
  • Adicionar Globals: Header e Footer editáveis pelo admin
  • Criar collection Posts com rich text (Lexical nativo do Payload)
  • Configurar SEO por página (title, description, og:image)
  • Implementar sistema de temas — extrair theme.config.ts
  • Criar collection Forms + FormSubmissions para formulários dinâmicos
  • Configurar upload de mídia com Uploadthing (grátis)
  • Customizar visual do painel admin para virar Broto CMS (logo, cores)
  • Deploy no Vercel — conectar ao Neon Postgres
  • Primeiro cliente real: usar este projeto como base
FASE 3 Escala — sistemas + VPS conforme cresce
  • Criar collections de negócio para sistemas (Products, Orders, etc.)
  • Implementar multi-tenant — um CMS, múltiplos clientes isolados
  • Migrar banco do Neon para PostgreSQL próprio no VPS (Hetzner ~€5/mês)
  • Configurar CI/CD: GitHub Actions → deploy automático no VPS
  • Adicionar Redis para cache de queries frequentes
  • Criar CLI interno: broto new [nome-cliente] que gera projeto base
  • Documentar processo de criação de tema para novos devs

payload.config.ts — configuração central

src/payload.config.ts typescript 
import { buildConfig } from 'payload'
import { postgresAdapter } from '@payloadcms/db-postgres'
import { lexicalEditor } from '@payloadcms/richtext-lexical'
import { uploadthingStorage } from '@payloadcms/storage-uploadthing'

import { Pages } from './collections/Pages'
import { Posts } from './collections/Posts'
import { Media } from './collections/Media'
import { Users } from './collections/Users'
import { Header } from './globals/Header'
import { Footer } from './globals/Footer'

export default buildConfig({
  admin: {
    user: Users.slug,
    meta: {
      titleSuffix: '— Broto CMS',  // sua marca no painel
    },
  },
  collections: [Pages, Posts, Media, Users],
  globals: [Header, Footer],
  editor: lexicalEditor({}),
  db: postgresAdapter({
    pool: { connectionString: process.env.DATABASE_URI }
  }),
  plugins: [
    uploadthingStorage({
      collections: { media: true },
      options: { token: process.env.UPLOADTHING_SECRET },
    }),
  ],
  secret: process.env.PAYLOAD_SECRET || '',
  typescript: { outputFile: 'src/payload-types.ts' },
})
12 — suporte

Bloqueios comuns e como resolver

Erro de conexão com banco
Verifique se DATABASE_URI no .env.local tem ?sslmode=require no final. O Neon exige SSL.
Admin não abre
Certifique que o arquivo app/(payload)/admin/[[...segments]]/page.tsx existe. É o que roteia para o painel do Payload.
Tipos desatualizados
Rode pnpm payload generate:types sempre que alterar collections. Isso regenera o payload-types.ts.
Imagens não aparecem
Configure NEXT_PUBLIC_SERVER_URL corretamente. Em dev: http://localhost:3000. Em prod: sua URL real.
Build falha na Vercel
Adicione todas as variáveis de ambiente nas configurações do projeto na Vercel. Elas não sobem junto com o código.
Não sei o que fazer
Manda aqui. Resolve junto. Sem julgamento.
